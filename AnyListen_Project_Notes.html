AnyListen Project Notes

Purpose: a very simple iPhone audio loopback tool: route a selected audio input to the current iOS audio output.

Current status

The app builds and runs. Core loopback works. Input selection works. Output selection uses Apple's native iOS route picker, which now opens reliably but has some system-imposed limitations.

Work completed

Core audio

  • Implemented AVAudioSession with .playAndRecord.
  • Implemented AVAudioEngine loopback from input node to main mixer/output.
  • Added microphone permission handling and required NSMicrophoneUsageDescription.
  • Added background audio mode.

UI overhaul

  • Removed scrolling layout.
  • Added dark blue/grey background.
  • Reduced branding to a small bold “AnyListen” title.
  • Added compact input/output cards and a large ear/listen button.

Input handling

  • Added an input picker using AVAudioSession.availableInputs.
  • Selected input is persisted in UserDefaults.
  • If the selected input disappears, the app shows it as missing instead of falling back silently.

Safety

  • Warns before listening through the iPhone speaker.
  • Stops listening on real audio route changes.
  • Does not auto-restart after route/device changes.
  • Attempts to preserve iPhone Speaker output when the user selected it before tapping Listen.

Important decisions

  • Keep scope minimal: no gain/boost, recording, presets, EQ, or unnecessary features.
  • Use native iOS routing UI for output: output selection is handled by AVRoutePickerView, because iOS does not expose a general public API for arbitrary output-device selection.
  • Use direct app UI for input: input selection is handled directly through setPreferredInput, because that is available through public APIs.
  • Prioritize feedback safety: speaker output is allowed only after a warning, and route changes stop listening.
  • Keep project reproducible: project generation is managed by xcodegen and project.yml.

Known iOS limitations / confusing behavior

  • Output selection is system-controlled. The app can show Apple's route picker, but it cannot fully own or persist arbitrary output choices the way it can with inputs.
  • Output “missing device” logic is limited. When AirPods or another output disconnects, iOS immediately chooses another route such as iPhone. The app cannot reliably show “AirPods missing” for output, because the native route picker does not expose a persisted selected output identity.
  • “Receiver” means the iPhone earpiece. The app now labels this as iPhone Earpiece, which is clearer than iOS’s low-level route name.
  • Bluetooth input + iPhone speaker output may be constrained by iOS. The app uses overrideOutputAudioPort(.speaker) when appropriate, but iOS may still make route decisions depending on the selected Bluetooth profile and current hardware.
  • The output selector icon is native. Apple's AVRoutePickerView draws its own icon/control. Styling it heavily is fragile, so the app now uses the native icon button rather than a custom overlapping “Select” label.

Current files of interest

  • AnyListen/AnyListen/ContentView.swift — main SwiftUI UI.
  • AnyListen/AnyListen/AudioEngineManager.swift — audio session, engine, input selection, route handling, safety logic.
  • AnyListen/AnyListen/AudioRoutePicker.swift — native Apple output route picker wrapper.
  • AnyListen/project.yml — XcodeGen project configuration.
  • AnyListen/AnyListen/Info.plist — generated plist containing microphone permission and background audio entries.

Recommended next steps

  1. Test target real-world setup: Rode Wireless ME USB microphone as input and hearing aids/AirPods/Bluetooth as output.
  2. Verify route labels: confirm input/output names are understandable for the intended user.
  3. Test safety paths: unplug selected input while listening, disconnect Bluetooth output, select iPhone Speaker, and confirm behavior is safe.
  4. Consider feedback detection later: possible but non-trivial. It may require signal analysis, thresholds, and careful CPU/battery tradeoffs.
  5. Consider an onboarding note: a one-time short note could explain that output selection is controlled by iOS and the route icon opens the iPhone output menu.

xcodegen generate

  xcodebuild -project AnyListen.xcodeproj -scheme AnyListen -sdk iphonesimulator build

DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer \
  xcodebuild -project AnyListen.xcodeproj -scheme AnyListen \
  -destination 'generic/platform=iOS' CODE_SIGNING_ALLOWED=NO build

Summary

AnyListen is currently best understood as a minimal public-API iOS audio loopback app. Input control can be app-owned. Output control must mostly remain iOS-owned. The biggest remaining product challenge is communicating that limitation clearly while keeping the app simple and safe.
